= Step 3 Gather Your Network Info & Configure Ansible 

What is Ansible?

If you are new to Ansible, here is a very brief overview.

Ansible is a tool used to automate tasks on computers.
Imagine you have a bunch of computers, and you want to make sure that they all have the same software installed. 
Maybe you need to create new user accounts on all of them. 
Instead of doing each task manually on every computer, taking tons of time and risking errors on some systems due to typos, you can use Ansible.
You can test your changes on one machine and get it right, then confidently apply the changes to all computers at once knowing that each system will the same.

The runtime components and a plain text file called an ansible playbook is used to automate tasks, like setting up a new user or installing software.
The idea of Ansible *playbook* comes from a sports play book analogy.
It can contain or can be passed variables, these get used by modules to perform tasks.
The variables can be thought of as the ball or puck in the game.
The modules are the players.
The tasks are the steps they need to take with the variables to score a goal.

=== The Ansible Inventory File

There is another component of running the ansible automation play that we need to look at called the inventory file.

An Ansible inventory file is a list of the computers (or hosts) that you want to manage with Ansible.
It's primary purpose is to define the available systems and the ways that you want to group them for plays.
It can also define how you connect to each system or group.
This is important because you often want to run the same tasks on multiple computers, but not all systems have the same attributes.
The inventory file helps you organize and manage your computers efficiently.

Ours is very simple. Let's take a look at it.

[source,bash,run]
----
cat  hosts.ini
----

This is the content of the file.

[source,ini]
----
localhost  ansible_connection=local
----

In this example, there are only two components to the line.

[arabic]
. *localhost*:
* the name of the computer. 
  By specifying `+localhost+` we are telling ansible that the local machine is a valid inventory member.
  We can execute playbooks against the local machine.
. *ansible_connection=local*:
* This part tells Ansible how to connect to the computer. 
`+local+` means that Ansible should run the tasks directly on the same computer where Ansible is running, without using a network connection.

=== Why would we specify connection=local?

This line is useful when you want to test your Ansible playbooks on your own computer before running them on other computers.
This eliminates connectivity issues as a source of problems when debugging your ansible.
We only have to concentrate on the changes we are making.
It’s like practicing your recipe in your own kitchen before cooking for a big party.
This helps you make sure everything works correctly before you use it on a bigger scale.


=== How do we pass information to the playbook?

Many times, all the data we need isn't in the playbook, or we want to override it.
We don't want to provide it in plaintext as part of the playbook file and we don't want it to be clear text on the command line.
It needs to be a secret. 
What do we do? 

Well, Ansible provides a several of ways. 
One way is to have ansible ask us at runtime for the plaintext password as a hidden field. 
This allows us to give the playbook additional configuration information to complete its tasks without storing it in the playbook or showing in our history.

There is a better way though. 
We use an encrypted file to keep the value secret.
These are called vault files.
Ansible provides a the ansible-vault executable to perform the encyption and decryption.
ansible-vault can be called automatically by ansible when it discovers an encrypted vault file.

Let's use the temporary password value `+redhat+`.
We will store it in a vault file.
It starts with a simple text file.

[source,bash,run]
----
echo 'tmp_password: "redhat"' > vault.txt
----

Let's encrypt the file now with ansible-vault.
Running it prompts us for a password and to verify the password.
The same passord is required to decrypt.
We can encrypt the file simply by calling the tool like this:

[source,bash,run]
----
ansible-vault encrypt vault.txt
----
Provide a simple password of your choice for this exercise.

Now run can safely run the playbook without exposing the temporary password. Let's apply the ansible playbook to the localhost.

Specify the inventory with `--inventory` or `-i` to be brief.

We have allowed the playbook to load additional variables by reading a filename passed as a variable.
To do this, use the commandline parameter `+--extra-vars+` or `+-e+` for short.
This is how you will specify the vault file.
Ansible will automatically detect that it is encrypted.

To specify the vault password so that ansible can read the encrypted data from vault.txt, you will add `+--ask-vault-password+`.
When you run the playbook, you will be prompted for the vault password. 

OK, pulling it all together we run:

[source,bash,run]
----
ansible-playbook --inventory hosts.ini --extra-vars "vault_path=vault.txt" --ask-vault-password add-audit-user.yml
----

Your results should look something like this:

=== Expanding your Automation

Edit the `+hosts.ini+` file to add the `+clients+` group.
This tells ansible that you now have your 'productions' systems available as targets to execute playbooks against.
Remember, you can use this same inventory file as the list of hosts for any playbook.

[source,bash,run]
----
cat > hosts.ini << EOF
localhost  ansible_connection=local
[clients]
client1
EOF
----

The `+hosts.ini+` should contain the `+clients+` group.

Run the playbook to apply our changes to the clients group. 
By specifying the `+--limits+` parameter, we can have ansible only target our clients group.
The limit can specify a group or individual host in the inventory file.

[source,bash,run]
----
ansible-playbook -i hosts.ini --limit clients --extra-vars "vault_path=vault.txt" --ask-vault-password add-audit-user.yml
----

You can test the client1 system in the same way that you tested the local system.

* SSH to the client1 system as root
* su with a login shell to the rhel user
* now try to su with a login shell as johnp, you should be prompted for the password
* verify that you can login

SUCCESS!

We have learned to do the following:
- run a playbook against a local machine and a remote machine.
- provide a list of hosts that we want to automate through the inventory file.
- filter that list using the --limit argument.
- provide extra variables at the commandline and how to load them in a playbook through a file.
- create secrets and store them in a vault file and use them a execution time.
